## Preflight: TARGET_PROJECT 解析与校验

在收集代码、日志、git 历史之前，先执行：

1. 从控制面板根目录 `AGENTS.md` 读取 `TARGET_PROJECT`
2. 将其解析为实际代码仓库根目录
3. 校验目录存在
4. 向用户回显最终解析路径
5. 后续所有代码阅读、日志排查、git 历史、测试和修复都在 `TARGET_PROJECT` 中进行

如果问题只涉及控制面板文档或 workflow 本身，则留在控制面板仓库处理；否则默认 bug 调查发生在 `TARGET_PROJECT`。

当遇到 bug、异常行为或性能问题时，按以下流程系统化排查：

1. **信息收集**
   - 错误信息（完整的 error message / stack trace）
   - 复现步骤（什么操作触发了问题）
   - 期望行为 vs 实际行为
   - 最近的代码变更（在 `TARGET_PROJECT` 中检查 `git log -5 --oneline`）

2. **形成假设**
   - 列出 3-5 个可能的原因
   - 按可能性从高到低排序
   - 标注每个假设需要检查什么

3. **逐一验证**
   对每个假设：
   - 设计一个最小验证实验
   - 运行验证 → 记录结果
   - 排除或确认 → 进入下一个

4. **修复并防范**
   - 修复根因（不是症状）
   - 添加测试覆盖该 bug 场景
   - 记录到 `.vibe/artifacts/error_journal.md`
   - 说明如何防止同类问题再现

5. **输出格式**

   ```markdown
   ## 🔍 Debug: [问题简述]

   ### 1. 症状
   [具体发生了什么]

   ### 2. 信息收集
   - Error: `[错误信息]`
   - File: `[文件路径]`
   - 触发条件: [什么操作触发]

   ### 3. 假设列表
   1. ❓ [最可能的原因]
   2. ❓ [次可能的原因]
   3. ❓ [较低可能的原因]

   ### 4. 调查过程
   **假设 1:** [检查了什么] → [结果]
   **假设 2:** [检查了什么] → [结果]

   ### 5. 根因
   🎯 [为什么会发生]

   ### 6. 修复
   [修复代码]

   ### 7. 防范
   🛡️ [如何预防同类问题]
   ```

> **关键原则**：问清楚再下手、测试假设不瞎猜、解释原因不只修表面、防止复发。
